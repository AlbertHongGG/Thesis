# 人工驗收確認表（Manual QA Checklist）

> 範圍：個人記帳＋月報表網站（Personal Expense Tracking & Monthly Report System）
> 目的：逐項勾選確認流程正確、權限邊界正確、資料隔離正確、列表/統計/圖表/匯出一致、以及全站狀態（Loading/Error/Empty/Retry）與錯誤碼處理（401/403/404/5xx）符合 Spec。

---

## 0. 環境 / 前置條件（Environment & Test Data Setup）

- [ ] 準備至少 2 組可登入帳號（Email+密碼），例如：`user1@example.com`、`user2@example.com`（用於資料隔離與跨帳號驗證）
- [ ] 確認密碼規則：註冊密碼至少 8 字元；驗收時需準備「<8 字元」的測試密碼
- [ ] 準備測試用「月份資料」：至少包含 2 個不同月份（例如當月與上月），確保月報表切換有資料可驗證
- [ ] 準備測試用「跨日」資料：至少 3 個不同日期（同一月份內），讓日分組與每日長條圖可驗證
- [ ] 準備測試用「類別分布」資料：至少 2 個支出類別（例如 食物/交通）與 1 個收入類別（例如 薪水），讓圓餅圖與統計可驗證
- [ ] 針對同一天準備「同日多筆」資料：至少 2 筆（用於同日內時間倒序驗證與當日總計驗證）
- [ ] 確認系統已建立 Spec 的預設類別（食物/生活/交通/薪水/提款），且可被停用但不可刪除
- [ ] 確認「停用類別」後，該類別不再出現在新增帳務可選清單（但歷史帳務仍顯示該類別名稱）
- [ ] 測試瀏覽器至少覆蓋：Chrome（桌機）與手機尺寸（<768px 的 RWD）
- [ ] 若系統有分頁/無限捲動（每頁 30 筆）：準備至少 35 筆帳務資料以驗證載入更多

## 1. 角色與權限邊界（Role & Access Boundary）

### 1.1 Guest（未登入）路由與導向

- [ ] 未登入直接開啟 `/transactions`：應導向 `/login`（UI）且受保護 API 回 401
- [ ] 未登入直接開啟 `/reports`：應導向 `/login`（UI）且受保護 API 回 401
- [ ] 未登入直接開啟 `/categories`：應導向 `/login`（UI）且受保護 API 回 401
- [ ] 未登入開啟 `/login`：可正常看到登入表單
- [ ] 未登入開啟 `/register`：可正常看到註冊表單

### 1.2 User（已登入）路由與導向

- [ ] 已登入狀態開啟 `/login`：應自動導向 `/transactions`（避免看到登入表單）
- [ ] 已登入狀態開啟 `/register`：應自動導向 `/transactions`
- [ ] 登出後嘗試使用瀏覽器返回回到受保護頁：應再次導向 `/login`（不可看到舊資料快照）

### 1.3 資料隔離（IDOR / user_id Boundary）

- [ ] `user1` 僅能讀取自己的 Transaction 與 Category：嘗試用 URL / API 存取 `user2` 的資料（以 id 指定）必須失敗（403 或 404，需與系統策略一致且不可洩漏他人資料）
- [ ] `user1` 建立的自訂類別，不應出現在 `user2` 的類別清單/新增帳務類別下拉中
- [ ] 系統預設類別（user_id 為 Null 的共用類別）應同時出現在 `user1` 與 `user2` 中

## 2. 端到端主流程（End-to-End Main Flows）

### 2.1 Guest 端到端流程（對應 User Flow 3.1）

- [ ] 以 Guest 開啟網站：預設進入登入頁（`/login`）
- [ ] 從登入頁點擊「註冊」連結可進入 `/register`
- [ ] 註冊：輸入合法 Email + 合法密碼 + 確認密碼相符，送出後註冊成功並自動登入
- [ ] 註冊成功自動導向 `/transactions`，且 Header 導覽切換為 User 狀態（顯示：帳務列表/月報表/類別管理/登出）
- [ ] 直接登入：在 `/login` 以正確帳密登入成功後導向 `/transactions`

### 2.2 User 主要流程：記帳（對應 User Flow 3.2 / 主要流程）

- [ ] `user1` 登入後進入 `/transactions`，列表初始載入成功並依日期分組顯示
- [ ] 點擊「新增帳務」開啟新增帳務表單（Modal）
- [ ] 表單填寫：type（income/expense）、amount（>0）、category（僅可選 active 類別）、date（YYYY-MM-DD）、note（<=200）
- [ ] 送出新增：成功後 Modal 關閉，新帳務出現在正確日期分組中，且同日總收入/總支出更新
- [ ] 編輯帳務：從列表點擊編輯，Modal 預填原資料；修改任一欄位後送出，列表與統計即時更新
- [ ] 編輯跨日期：將帳務日期改到其他日，帳務應移動到對應日期分組，且兩個日期的日統計皆正確更新
- [ ] 刪除帳務：點擊刪除顯示二次確認；確認刪除後列表移除該筆，且日統計/月統計/圖表（若在同月）同步更新

### 2.3 User 次要流程：查看月報表（對應 User Flow 3.2 / 次要流程）

- [ ] 從 Header 進入 `/reports`，預設顯示當前年月報表
- [ ] 統計卡片顯示：總收入、總支出、淨收支（計算正確且與當月原始帳務加總一致）
- [ ] 切換年份/月份：切換後重新載入，顯示該年月的統計與圖表
- [ ] 圓餅圖：僅統計支出（expense），類別分組金額與占比可被核對（加總應等於總支出）
- [ ] 長條圖：每日顯示收入柱與支出柱；柱狀金額與該日帳務加總一致

### 2.4 User 次要流程：類別管理（對應 User Flow 3.2 / 次要流程）

- [ ] 從 Header 進入 `/categories`，載入類別清單成功
- [ ] 可新增自訂類別：新增後立即出現在類別清單，且可在新增帳務時被選取（若 active）
- [ ] 可編輯類別名稱：編輯後清單更新，且歷史帳務顯示名稱需與更新後一致（若系統設計為引用同一類別紀錄）
- [ ] 可停用類別（is_active=false）：停用後不應出現在新增/編輯帳務類別下拉清單
- [ ] 停用類別後，歷史帳務仍顯示該類別（不可變更為空/unknown）
- [ ] 預設類別不可刪除（若 UI 有刪除入口需不存在或操作被拒絕）

### 2.5 User 次要流程：匯出資料（CSV）（對應 User Flow 3.2 / 次要流程，選擇性）

- [ ] 若已實作匯出：在 `/reports` 點擊「匯出 CSV」會下載檔案
- [ ] 若已實作匯出：檔名符合 `transactions_YYYY_MM.csv`
- [ ] 若已實作匯出：CSV 欄位包含「日期、類型、類別、金額、備註」
- [ ] 若已實作匯出：匯出資料集合與畫面顯示一致（同年月、同篩選條件；列數與加總可核對）

## 3. 全站狀態品質（Loading / Error / Empty / Retry）

### 3.1 表單與動作 Loading（防重送）

- [ ] 登入送出時顯示 Loading，且按鈕 disabled（或等效）避免重複提交
- [ ] 註冊送出時顯示 Loading，且按鈕 disabled 避免重複提交
- [ ] 新增/編輯帳務送出時顯示 Loading，且避免重複送出造成重複新增
- [ ] 刪除帳務確認時顯示 Loading，且避免重複刪除造成錯誤狀態
- [ ] 新增/編輯類別送出時顯示 Loading，且避免重複送出
- [ ] 停用/啟用類別時顯示 Loading，且避免連點造成狀態翻轉錯亂
- [ ] 匯出 CSV（若有）觸發下載時，UI 有明確回饋（下載中/已開始下載）且不會連點觸發多次下載（若有設計）

### 3.2 清單/頁面載入 Error 與 Retry

- [ ] `/transactions` 載入失敗（5xx/網路錯誤）時顯示 Error 狀態並提供 Retry
- [ ] `/reports` 載入失敗時顯示 Error 狀態並提供 Retry
- [ ] `/categories` 載入失敗時顯示 Error 狀態並提供 Retry
- [ ] Retry 會重新發送對應 API 請求且成功後回到 Ready

### 3.3 Empty 狀態

- [ ] `/transactions` 在無任何帳務時顯示 Empty 狀態，並引導新增第一筆帳務
- [ ] `/reports` 在該月無帳務時顯示 Empty 狀態（統計為 0 或等效呈現，且圖表顯示空資料提示）
- [ ] `/categories` 若無自訂類別仍能呈現預設類別，且不會變成完全空白頁

---

## 4. 錯誤碼與導向（401 / 403 / 404 / 5xx）

- [ ] 任一受保護 API 回 401（Token 過期/未登入）時：前端導向 `/login`，並顯示「請重新登入」等可理解提示
- [ ] 403（無權限存取他人資源）時：顯示 Forbidden（或收斂為 404），且不得洩漏他人資料存在性
- [ ] 404（資源不存在或路由不存在）時：顯示 Not Found
- [ ] 5xx 時：顯示 Error 狀態並提供 Retry；不得使 UI 停留在永遠 Loading

## 5. 功能需求驗收（Functional Requirements Driven）

### 5.1 帳號與認證（Authentication）（對應 4.1）

- [ ] 註冊 Email 格式驗證：輸入不合法 Email 會顯示欄位錯誤並禁止送出
- [ ] 註冊密碼長度驗證：密碼 < 8 字元會顯示錯誤並禁止送出
- [ ] 註冊確認密碼驗證：確認密碼不一致時顯示錯誤並禁止送出
- [ ] Email 唯一性：使用已存在 Email 註冊會收到明確錯誤（例如 409）且 UI 顯示「Email 已被使用」
- [ ] 登入成功後可取得登入狀態（Session/Token 生效），並能進入受保護頁（`/transactions`）
- [ ] 登入失敗（帳密錯誤）顯示明確錯誤，且 Email 不被清空（密碼可視設計清空）
- [ ] 登出後清除 Session/Token，並導向 `/login`
- [ ] Session 過期：在任一受保護頁觸發 API 時回 401，應被導向 `/login`
- [ ] 資料綁定 user_id：新增 Transaction/Category 後，在資料層確認其 user_id 為當前登入者（不可被前端竄改）

### 5.2 帳務紀錄（Transaction）（對應 4.2）

- [ ] 新增帳務必填欄位驗證：type/amount/category/date 缺任一項時不得送出，且欄位錯誤提示清楚
- [ ] amount 驗證：輸入 0、負數、非整數時不得送出，且提示明確
- [ ] date 格式驗證：非 YYYY-MM-DD 格式不得送出（或送出後被 API 拒絕並顯示錯誤）
- [ ] note 長度驗證：超過 200 字不得送出（或送出後被 API 拒絕並顯示錯誤）
- [ ] category 有效性：只能選擇 active 類別；對已停用類別新增應被 UI 禁止或被 API 拒絕
- [ ] 編輯可修改所有欄位：修改後列表立即更新
- [ ] 刪除需二次確認：取消則不刪除；確認才刪除
- [ ] 交易後一致性：新增/編輯/刪除後，`/transactions` 的日分組、日總計、`/reports` 的月統計與圖表（同月）皆同步一致
- [ ] 跨帳號隔離：`user1` 無法編輯或刪除 `user2` 的帳務（UI 無入口 + API 拒絕）

### 5.3 類別管理（Category）（對應 4.3）

- [ ] 類別名稱必填且 <= 20 字：不符合時不得送出，提示清楚
- [ ] 類別唯一性（同使用者內唯一）：新增/編輯成重複名稱會被拒絕（顯示明確錯誤）
- [ ] 類別 type 欄位可用（income/expense/both）：新增後在新增帳務時可正確出現（若系統有依 type 篩選，需符合設計）
- [ ] 類別停用（is_active=false）後：不出現在新增帳務類別選擇
- [ ] 歷史保留：類別停用後，已存在 Transaction 仍顯示該類別名稱
- [ ] 類別不可刪除：若嘗試刪除（UI/API）應被拒絕或 UI 不提供入口
- [ ] 預設類別可停用：停用後也不出現在新增帳務可選清單
- [ ] 跨帳號隔離：`user2` 看不到 `user1` 的自訂類別

### 5.4 主要頁面需求與導覽規則（Page Requirements）（對應 4.4）

- [ ] Header 固定在頁面頂部，包含 Logo，點擊 Logo 導向 `/transactions`
- [ ] Guest Header 僅顯示登入/註冊相關導覽；不顯示帳務列表/月報表/類別管理/登出
- [ ] User Header 顯示帳務列表/月報表/類別管理/登出；不顯示登入/註冊
- [ ] CTA 去重：登入入口不與頁面 CTA 重複；新增帳務僅在 `/transactions`；匯出 CSV 僅在 `/reports`
- [ ] `/transactions` 的新增帳務入口可被清楚找到（FAB 或頁面按鈕）
- [ ] `/reports` 有年月選擇器並可切換到當年與前 2 年

### 5.5 帳務列表頁（日分組顯示）（對應 4.5~4.6）

- [ ] 列表依日期分組，最新日期在最上方
- [ ] 每個日期區塊顯示「日期標題、當日總收入、當日總支出」
- [ ] 同日內帳務依時間倒序（最新在前）
- [ ] 顯示金額規則一致：收入與支出在 UI 上可被清楚區分（顏色或符號；以實作為準但需一致）
- [ ] 即時更新：任一 CRUD 後，該日總收入/總支出立即更新

### 5.6 月報表（Monthly Report）（對應 4.7）

- [ ] 預設顯示當前年月
- [ ] 切換年份/月份會重新載入並更新統計卡片與圖表
- [ ] 統計計算正確：總收入/總支出/淨收支與當月原始帳務加總一致
- [ ] 該月無帳務時：統計顯示為 0（或等效）且顯示空資料提示

### 5.7 圖表需求（Charts）（對應 4.8）

- [ ] 圓餅圖只統計支出（expense）；其分組加總等於總支出
- [ ] 圓餅圖空資料時顯示「本月無支出」
- [ ] 長條圖 X 軸為日期（1~31，僅顯示有資料日期或依實作一致）；Y 軸為金額
- [ ] 長條圖每日收入/支出柱狀值可被核對（等於該日帳務加總）
- [ ] 長條圖空資料時顯示「本月無資料」

### 5.8 匯出 CSV（選擇性）（對應 4.9）

- [ ] 若已實作：匯出只包含當前選擇月份資料
- [ ] 若已實作：匯出欄位順序/內容符合 Spec（日期、類型、類別、金額、備註）
- [ ] 若已實作：匯出後用加總核對（CSV 加總 vs 畫面統計 vs 圖表加總）一致

## 6. 非功能需求驗收（Non-functional Requirements Driven）（對應 5）

- [ ] RWD：桌機/平板/手機尺寸都能完成核心流程（登入/新增帳務/查看報表/管理類別），不需橫向捲動才能操作
- [ ] 手機版 Header 會收合成漢堡選單，且能正常導覽到 `/transactions`、`/reports`、`/categories` 與登出
- [ ] Loading/Empty/Error 狀態呈現一致且可理解（Skeleton/Spinner、提示文字、Retry）
- [ ] 錯誤提示可理解：欄位層級與表單層級訊息不互相矛盾
- [ ] 401/403/404/5xx 行為符合 Spec 的錯誤處理表
- [ ] 資料一致性：同一月份的列表/統計/圖表/（若有）匯出使用相同資料集合；CRUD 後即時同步
- [ ] 效能：當帳務很多時（例如 >30 筆），分頁/無限捲動可用且不會卡頓（以體感+網路請求數可觀測）
- [ ] 安全性：密碼不會以明文回傳；登入/註冊錯誤訊息不洩漏敏感資訊
- [ ] XSS：在 note 或 category name 輸入 HTML/Script 字樣不會被當成可執行內容（應被轉義/安全顯示）
- [ ] CSRF：對需要認證的 API，不帶有效 Token 應被拒絕（401）
- [ ] UI 一致性：同一動作入口在全站一致；Header 與頁面內不重複相同 CTA（登入/新增帳務/匯出）
