### **質數路徑（Prime Path）**

https://www.researchgate.net/publication/391953357_Prime_Path_Coverage_in_the_GNU_Compiler_Collection

質數路徑的概念旨在解決迴圈造成的路徑無限問題。質數路徑要求測試案例必須覆蓋進入迴圈、執行迴圈體至少一次以及跳過迴圈的路徑情況。專注在控制流圖中的執行路徑序列與迴圈結構。

### 定義

質數路徑被定義為「一條 Simple Path，且它不是任何其他 Simple Path 的真子路徑（Proper Subpath）」。   

### **理論優勢**

質數路徑覆蓋在發現缺陷的能力上，往往優於邊對覆蓋（Edge-Pair Coverage），並且在某些條件下包含（Subsume）了 MC/DC 標準，因為它能捕捉到**數據依賴導致的不可行路徑（Infeasible Paths）**。

### **演算法流程詳解**

1. **控制流圖分解（CFG Decomposition）**：首先將大型的 CFG 分解為強連通分量（Strongly Connected Components, SCC）。由於質數路徑不會跨越非連通的結構，這種分治策略（Divide and Conquer）顯著縮小了問題規模。
2. **路徑枚舉與後綴樹構建**：在每個 SCC 內部，算法並非盲目生成路徑，而是將已發現的簡單路徑插入到一棵後綴樹中。
3. **冗餘剪枝（Redundancy Pruning）**：當演算法探索一條新路徑時，會查詢後綴樹。如果該路徑已經是樹中某條現有路徑的子路徑，則該路徑被標記為冗餘並停止擴展；反之，如果新路徑是現有路徑的超集（Superpath），則更新樹結構。這種機制避免了對大量非質數路徑的無效存儲與比較。

### 優化

採用了位元運算（Bitwise Operations）來追蹤路徑索引，利用位元圖（Bitmap）的緊湊性來表示大量的路徑集合。這種位元級別的優化極大降低了記憶體消耗，使得在普通開發工作站上對數百萬行代碼進行質數路徑分析成為可能

### **靜態分析：程式碼屬性圖（CPG）**

https://arxiv.org/html/2507.16585v1

一種革命性的中間表示法（Intermediate Representation）旨在將程式的多重維度融合為單一的圖結構 。 

CPG 整合了三種經典的圖表示：

1. **抽象語法樹（AST）**：捕捉程式的語法結構與層次關係。
2. **控制流圖（CFG）**：描述語句之間的執行順序與轉移邏輯。
3. **程式依賴圖（PDG）**：包含數據依賴（Data Dependency）與控制依賴（Control Dependency），描述變數如何影響運算結果。

在 CPG 中，節點（Nodes）代表程式碼元素（如方法、變數、字面量），而邊（Edges）則代表上述三種圖的關係。這種融合使得分析師可以通過圖遍歷來回答複雜的語義問題，例如：「是否存在一條路徑，使得使用者輸入（Source）在未經淨化（Sanitization）的情況下流向了 SQL 執行函數（Sink）？」。

### **可達性分析**

```scala
def getFlow() = {

```

為了避免全圖遍歷導致的爆炸，現代工具採用了「切片（Slicing）」技術，僅保留與特定變數或控制流相關的子圖進行分析 。

### **RepoScope 與 RSSG：專案層級的語義導航**

https://www.researchgate.net/publication/384210975_Iterative_Refinement_of_Project-Level_Code_Context_for_Precise_Code_Generation_with_Compiler_Feedback

https://arxiv.org/html/2507.14791v2

為了更精確地處理儲存庫層級（Repository-Level）的路徑分析，最新的研究提出了 RepoScope 框架及其核心數據結構——儲存庫結構語義圖（Repository Structural Semantic Graph, RSSG）。

- **RSSG 的構建**：不同於傳統 CPG 僅關注代碼指令，RSSG 納入了檔案結構、命名空間、類別繼承以及文檔註釋等高層語義資訊。它是一個異質圖（Heterogeneous Graph），能夠更全面地捕捉軟體架構的意圖。
- **呼叫鏈感知（Call Chain-Awareness）**：RepoScope 引入了呼叫鏈預測演算法。利用 RSSG 中的結構語義，該演算法能夠預測目標函數潛在的呼叫者與被呼叫者，即使在動態語言中也能以較高準確率重建斷裂的呼叫路徑。這對於在大型專案中找出跨模組的轉移路徑至關重要 。

### **混合分析架構（Hybrid Program Analysis）**

https://arxiv.org/html/2503.13580v1

https://arxiv.org/pdf/2503.13580

Panta 是一個具代表性的現代測試生成框架，它展示了如何將傳統演算法與 LLM 結合以解決路徑覆蓋問題 。Panta 摒棄了單純依賴 LLM 生成測試的「盲測」模式，採用了混合策略：

1. **靜態控制流分析**：首先，利用靜態分析工具計算出待測函數的控制流圖（CFG），並基於環形複雜度（Cyclomatic Complexity）識別出所有線性獨立的路徑。這些路徑構成了測試生成的「目標集合」。
2. **動態覆蓋率反饋**：系統生成初始測試案例並執行，收集動態覆蓋率數據（如 Jacoco 報告）。
3. **迭代提示（Iterative Prompting）**：將「未被覆蓋的路徑」資訊轉化為自然語言提示（Prompt），明確指示 LLM 生成能觸發特定分支條件的輸入。例如：「目前的測試未覆蓋 `if (x < 0)` 的分支，請生成一個 `x` 為負數的測試案例。」

相比僅依賴 LLM 的方法或傳統的搜索式測試生成

- **行覆蓋率（Line Coverage）**：提升了 26%。
- **分支覆蓋率（Branch Coverage）**：提升了 23%。 這種「靜態指引 + 動態反饋 + LLM 生成」的閉環機制，有效地解決了 LLM 容易忽略邊界條件與深層路徑的問題 。

### **PALM：路徑感知的符號執行互動**

https://arxiv.org/html/2506.19287v1

PALM 系統則聚焦於解決符號執行（Symbolic Execution）中的路徑理解與驗證問題 。

- **符號樹可視化**：PALM 設計了一個互動式前端，將後端的符號執行樹（Symbolic Execution Tree）可視化呈現。開發者可以直觀地看到每一條路徑的約束條件（Path Constraints）。
- **LLM 作為約束求解助手**：當符號執行引擎遇到難以求解的約束（例如涉及複雜字串操作或雜湊函數）時，PALM 會將該約束傳遞給 LLM。LLM 利用其對代碼庫的知識，嘗試生成滿足約束的具體輸入（Concrete Input）。
- **路徑驗證**：PALM 允許使用者指定樹上的一條特定路徑，並要求 LLM 生成專門覆蓋該路徑的測試。系統隨後會自動執行該測試，驗證其實際執行軌跡是否與目標路徑吻合，從而確保生成的準確性。

### **CodexGraph：基於 Agent 的多跳推理**

https://www.emergentmind.com/topics/codexgraph

在處理全專案層級的路徑分析時，單次檢索（RAG）往往無法獲取足夠的上下文。CodexGraph 提出了一種基於 Agent 的多跳推理（Multi-hop Reasoning）機制 。

- **雙 Agent 工作流**：系統包含兩個主要的 Agent。一個負責將自然語言問題分解為邏輯查詢步驟；另一個負責將這些步驟轉化為具體的圖數據庫查詢語言（如 Cypher）。
- **迭代式圖遍歷**：Agent 不會一次性獲取所有資訊，而是像人類專家一樣，在 CPG 上進行探索。它首先查詢函數的定義，然後追蹤其呼叫者，接著分析呼叫者的數據流。這種步進式的推理過程，使得 LLM 能夠在數百萬行代碼中精確定位出一條跨越多個檔案的複雜轉移路徑，有效克服了上下文窗口的限制。

### 專有名詞

- Simple Path
    
    在圖論中，如果一條路徑不包含重複的頂點（除了可能的起點與終點相同以形成簡單環），則稱之為簡單路徑（Simple Path）。
    
- 邊對覆蓋（Edge-Pair Coverage）
    
    這是一種結構化測試標準，要求測試案例必須執行控制流圖中所有長度為 2 的路徑（即由兩個連續邊組成的序列）。
    
    - **概念**：如果程式從節點 A 走到 B，再從 B 走到 C，這就是一個「邊對」。
    - **目的**：它比單純的「邊覆蓋（Edge Coverage，即分支覆蓋）」更嚴格，因為它不僅確保每個分支被執行，還確保了「分支之間的連接」被測試到。例如，它能檢測出「進入某個節點的方式」是否會影響「離開該節點的行為」。
- MC/DC（Modified Condition/Decision Coverage，修訂條件/決策覆蓋）
    - **核心定義**：在一個複雜的邏輯判斷（例如 `if (A and B) or C`）中，MC/DC 要求證明**每一個**條件（A、B、C）都能**獨立地**影響整個判斷的結果。
    - **作法**：你需要設計測試案例，使得當改變條件 A 的真假值（由 True 變 False），而其他條件（B、C）保持不變時，整個 `if` 的結果也會隨之改變。這避免了測試案例雖然覆蓋了 True/False 結果，但實際上是被其他條件「掩蓋（Masking）」的情況 。
- 真子路徑（Proper Subpath)
    - **定義**：如果路徑 $p$ 是路徑 $q$ 的一部分（子序列），且 $p$ 的長度小於 $q$（即 $p$ 不等於 $q$），則稱 $p$ 為 $q$ 的「真子路徑」。
    - **在演算法中的意義**：質數路徑被定義為「一條簡單路徑，且它不是任何其他簡單路徑的真子路徑」。這意味著質數路徑是「盡可能長」的路徑，如果再往兩端延伸就會造成節點重複（變成環）或無法延伸。這確保了測試會盡可能完整地遍歷迴圈與分支鏈，而不會陷入無限迴圈 。
- 數據依賴導致的不可行路徑（Infeasible Paths）
    
    能協助識別出那些「雖然語法結構上存在，但邏輯上永遠跑不到」的路徑。
    
    - **什麼是不可行路徑**：假設程式碼結構允許路徑 A -> B -> C。但如果 A 節點設定了 `x = 0`，而 C 節點的進入條件是 `if (x > 5)`。從控制流圖（CFG）的形狀看，這條路徑是存在的；但因為變數 `x` 的數據依賴關係（Data Dependency），這條路徑在執行時是不可能的（Infeasible）。
    - **為何能捕捉**：較弱的覆蓋率（如分支覆蓋）可能只要求分別測過 `x=0` 和 `x>5` 的分支（在不同次執行中），因此不會發現這兩者「串在一起」時的矛盾。但「質數路徑覆蓋」或全路徑分析會要求測試「A -> B -> C」這條特定序列。當測試生成器試圖生成滿足此路徑的輸入數據時，會發現約束條件互相矛盾（Conflict），從而識別出這是死代碼或邏輯缺陷 。
- 控制流圖（Control Flow Graph, CFG）
    
    這是程式碼的一種圖形化表示，是所有路徑分析演算法的基礎。
    
    - **節點（Nodes）**：代表「基本區塊（Basic Block）」，即一段連續執行、中間沒有跳轉的程式碼。
    - **邊（Edges）**：代表控制流的轉移（如 `if` 跳轉、`while` 迴圈、函數呼叫）。
    - **作用**：它將程式碼轉化為數學上的圖，使得我們可以利用圖論演算法（如 DFS、BFS）來尋找路徑 。
- 後綴樹（Suffix Tree）
    
    這是一種用於處理字串或序列的高效資料結構，在最新的 GCC 質數路徑演算法中扮演關鍵角色。
    • **原理**：它將所有已知的路徑序列儲存成一棵樹，樹的每一條邊代表序列的一個片段。
    • **在路徑分析中的作用**：當演算法在圖中搜索出一條新路徑時，需要判斷它是否是已知路徑的「子路徑」。如果沒有後綴樹，需要與所有已知路徑逐一比對（速度極慢）。有了後綴樹，只需在樹上查詢即可快速判斷。這就是為什麼新演算法能將複雜度從 $O(n^2m^2)$ 降低到 $O(n^2m)$ 的關鍵 。
    
- **呼叫圖（Call Graph）**
    
    呼叫圖是一種有向圖（Directed Graph），用於抽象表示電腦程式中計算單元（Processes）之間的調用關係。形式上，一個呼叫圖 $G = (V, E)$ 由以下元素組成：
    
    - **節點（Nodes, $V$）**：代表程式中的每一個過程、函數、方法或進入點。在某些細粒度的分析中，節點也可以代表一個具體的「調用點（Call Site）」，即源代碼中發生調用的具體行號或指令位置。
    - **邊（Edges, $E$）**：代表調用關係。若函數 $f$ 在其執行過程中可能調用函數 $g$，則存在一條從 $f$ 指向 $g$ 的有向邊 $(f, g) \in E$。

### 基礎 transition path 測試作法

- **節點覆蓋 (Node Coverage)：** 要求測試路徑集合必須訪問圖中的每一個節點至少一次。這相當於代碼測試中的「語句覆蓋」。
    - *缺陷：* 極其膚淺。例如，它無法檢測「從 A 到 B 再到 C」的特定路徑錯誤，只能確認 A、B、C 都存在。
- **邊覆蓋 (Edge Coverage)：** 要求測試路徑集合必須執行圖中的每一條邊至少一次。這相當於「分支覆蓋」。
    - *缺陷：* 無法檢測依賴於上下文的錯誤。例如，一個變數在狀態 A 被設為 null，但在經過狀態 B 後才被使用。如果測試只覆蓋了邊，可能剛好錯過了這條特定的觸發路徑。
- **邊對覆蓋 (Edge-Pair Coverage)：** 覆蓋所有長度為 2 的路徑片段。這開始具備檢測交互錯誤的能力，但路徑數量開始激增。

### **全質路徑覆蓋 (Prime Path Coverage, PPC)**

> 限制了路徑的無限增長
> 

PPC 被學術界公認為解決路徑爆炸問題最有效的系統性方法之一 25。它在「全路徑覆蓋」（不可行）和「邊覆蓋」（太弱）之間取得了一個完美的數學平衡。

- **定義：**
    - **簡單路徑 (Simple Path)：** 一條路徑中沒有重複的節點（除了起點和終點可以相同，形成簡單迴圈）。
    - **質路徑 (Prime Path)：** 一條簡單路徑，且它不被包含在任何更長的簡單路徑中（它是極大的）。
- **減量效果：**
    - 它強制測試路徑必須進入迴圈、繞行迴圈、並跳出迴圈，但不會無限繞行。
    - 它在數學上保證了路徑數量是有限的，即使在有環的圖中也是如此 。

**PPC 的數學優勢：**

1. **有限性 (Finiteness)：** 即使圖中有無窮迴圈，簡單路徑的長度受限於節點總數（最長不超過 $|V|$），因此質路徑的數量是嚴格有限的。
2. **迴圈處理 (Loop Handling)：** 質路徑會自然地包含「進入迴圈」、「繞行迴圈一圈回到起點」以及「跳過迴圈」的路徑。這迫使測試案例必須驗證迴圈邊界條件。
3. **包含關係 (Subsumption)：** PPC 數學上包含（Subsumes）了節點覆蓋和邊覆蓋，這意味著滿足 PPC 的測試集一定也滿足前兩者 27。

### **減量與篩選**

### 作法

- **旁路剪枝 (Side-track Pruning) 與後綴樹**
在計算質路徑的過程中，許多路徑共享相同的子結構。如果使用暴力的深度優先搜尋（DFS），會產生大量的重複計算。
• **演算法優化：** 利用 **後綴樹 (Suffix Tree)** 或 **Trie** 結構來存儲已發現的路徑。當 DFS 正在探索一條路徑時，可以即時檢查當前路徑的後綴是否已經存在於樹中。如果是，則可以進行剪枝或合併。
- **測試套件最小化 (Test Suite Minimization)**
這是一個經典的 **集合覆蓋問題 (Set Cover Problem)**。我們有一組目標（所有的質路徑 $TR$），我們需要找到最小數量的測試路徑（$TP$），使得這些測試路徑能「遊歷」（Tour）所有的質路徑。
    - **遊歷 (Touring) 的定義：** 一條測試路徑 $p$ 遊歷一條質路徑 $q$，當且僅當 $q$ 是 $p$ 的子路徑。
    - **貪婪演算法 (Greedy Algorithm)：**
        1. 初始化：待覆蓋集合 $U = TR$，選定路徑集 $S = \emptyset$。
        2. 當 $U$ 不為空時：
            ▪ 在所有可能的測試路徑中，選擇一條路徑 $p$，它能覆蓋 $U$ 中最多的元素。
            ▪ 將 $p$ 加入 $S$。
            ▪ 從 $U$ 中移除被 $p$ 覆蓋的所有質路徑。
        3. 輸出 $S$。
- **基於相似度的聚類篩選 (Similarity-based Clustering)**
    
    如果有兩條路徑：`A -> B -> C -> D` 和 `A -> B -> E -> D`，它們高度相似。
    
    - **方法：** 計算路徑之間的字串相似度（如 Levenshtein Distance）或向量相似度（使用 CodeBERT 等模型 embedding）。
    - **篩選：** 將高度相似的路徑視為同一群組，從每個群組中只採樣一條代表性路徑進行測試，以減少重複測試相同邏輯的浪費 。
- **基於風險與權重的優先級篩選 (Risk-based Prioritization)**
除了結構上的覆蓋，我們還可以引入業務價值（Business Value）作為篩選維度。並非所有狀態都同等重要。
    - **權重分配：**
        - **使用頻率：** 根據生產環境的 Analytics 數據（如 Google Analytics），將使用者最常訪問的狀態和點擊的按鈕賦予高權重。
        - **缺陷密度：** 根據歷史 Bug 追蹤系統，將過去常出錯的模組賦予高權重。
        - **業務關鍵性：** 手動標記「結帳」、「支付」、「登入」等關鍵路徑。
    - **加權隨機遊走 (Weighted Random Walk)：** 修改圖遍歷算法，使其在選擇下一個節點時，不是均勻隨機，而是根據邊的權重機率進行選擇。這能生成更符合真實使用者行為分佈的測試路徑 31。
    - **遺傳演算法 (Genetic Algorithms)：** 將路徑生成視為多目標優化問題（覆蓋率 vs. 測試長度 vs. 風險權重）。利用遺傳算法演化出最優的路徑組合，這在大型工業級案例中表現優於貪婪算法 33。

### **減量（Reduction）**

> 從根本上減少需要探索的路徑總數。這類方法通常利用邏輯等價性，將多條路徑合併為一條，或證明某些路徑是多餘的而不予探索。
> 
- **狀態合併（State Merging）**
- **偏序縮減（Partial Order Reduction, POR）**
    
    執行結果已知與先前探索過的某條路徑直接跳過不予執行 
    
    https://www.researchgate.net/publication/220997929_Dynamic_partial-order_reduction_for_model_checking_software
    
- **程式切片（Program Slicing）**
    
    基於依賴分析的減量技術。如果目標是驗證程式中某個特定點（例如第 100 行的 `assert(x!= null)`）的安全性，那麼程式中所有不影響變數 `x` 的指令與分支路徑都是無關的 。
    
    https://ieeexplore.ieee.org/document/526534
    
    https://www.researchgate.net/publication/271880875_Abstract_program_slicing_on_dependence_condition_graphs
    
- **抽象解釋（Abstract Interpretation）的路徑修剪**
    
    利用數學估算變數的值域，若分支條件不可能滿足，則直接修剪該路徑。一種低成本的過濾機制，能夠在路徑生成的最前端攔截無效的 Transition Path 。   
    
    https://www.researchgate.net/publication/321097727_Leveraging_Abstract_Interpretation_for_Efficient_Dynamic_Symbolic_Execution
    

### **篩選與優先級策略**

- **基礎搜尋啟發式**
    - **隨機路徑搜尋（Random Path Search）**
        
        搜尋維護一棵執行樹，從根節點開始隨機選擇分支直到葉節點。這種策略能夠均勻地探索程式的各個部分，避免因某個複雜函數的迴圈而導致程式其他部分無法被測試 。
        
        https://www.researchgate.net/publication/395428067_Optimizing_Symbolic_Execution_Path_Exploration_with_a_Transfer_Learning-Based_Strategy
        
        https://www.researchgate.net/publication/311471839_Steering_symbolic_execution_to_less_traveled_paths
        
    - **覆蓋率優化搜尋（Coverage-Guided Search）**
        
        優先選擇那些能夠觸發新基本區塊（Basic Block）的路徑。
        
- **定向符號執行（Directed Symbolic Execution, DSE）**
    
    DSE 的目標不是遍歷全程式，而是盡快到達某個特定的目標點（Target），例如崩潰發生點或特定的惡意程式碼特徵。
    
    - **距離度量（Distance Metrics）**
        
        經典 DSE 使用控制流圖（CFG）上的最短路徑算法（如 Dijkstra）來計算每個狀態距離目標的距離。這作為優先級佇列的權重，距離越短權重越高 。
        
        https://arxiv.org/html/2508.21302v1
        
    - **調用鏈後向符號執行（Call-Chain-Backward SE, CCBSE）**
        
        從包含目標的函數開始，反向沿著調用鏈（Call Chain）向上尋找可行路徑。這種方法能夠迅速排除那些根本無法到達目標函數的路徑 。
        
        https://www.cs.tufts.edu/~jfoster/papers/thesis-ma.pdf

### 結合 GNN 的路徑優先級排序 (GNN-based Prioritization)

### **圖論優先級：最小路徑覆蓋（Minimum Path Cover）**

https://arxiv.org/html/2505.03555v1

為了達到程式碼覆蓋率，並不需要探索所有路徑，甚至不需要探索所有「有意義」的路徑，只需要探索一個「最小路徑集」即可 。

- **核心方法論**
    
    > 將路徑篩選問題建模為圖論中的「路徑覆蓋」問題。
    > 
    1. **iCFG 轉換與 DAG 分解**：首先將程式的程序間控制流圖（iCFG）轉換為多個有向無環圖（DAG）。
    2. **最小路徑覆蓋（MPC）計算**：在 DAG 上計算 MPC。MPC 是指能夠覆蓋圖中所有節點所需的最少路徑集合。將 DAG 轉化為二分圖並求解最大匹配（Maximum Matching）來解決。
    3. **運行時導引**：不再依賴隨機或距離啟發式，而是嚴格優先調度那些屬於預先計算出的 MPC 集合中的路徑。
- **數據實證與效能**
    
    > 這項研究證明了，傳統符號執行所探索的絕大多數 Transition Path 對於覆蓋率目標而言都是冗餘的。
    > 
    - **狀態縮減**：符號狀態數量減少了高達 **88.6%**。
    - **記憶體節省**：記憶體使用量減少了 **93.5%**。
    - **覆蓋率提升**：相較於 KLEE 的最佳搜尋策略，基本區塊覆蓋率提升了 **19.6%**，且發現了更多的安全違規（Security Violations）。

    透過 LLM 去根據語意標註每個節點的重要程度，來找出真正重要的 transition path

### Speckit 對於完善 C**onstitution 指令的構想**

```bash
/speckit.constitution

本專案為「真實上線之全端產品」，所有產出需以可維運性、可擴充性與長期穩定性為前提。
以下規範為強制性原則，未遵守者視為不合格輸出。

────────────────────────
【一、系統正確性與資料一致性（最高優先）】
────────────────────────
- 不可產生會導致資料不一致、狀態錯亂或競態條件的設計
- 所有狀態轉換需具備明確前置條件與後置驗證
- 跨系統、前後端互動必須定義清楚的資料契約（Schema / 型別）
- 對於可能失敗的操作，需設計可回復（rollback / compensation）機制
- 不可假設外部系統永遠可靠

────────────────────────
【二、程式碼品質與架構邊界】
────────────────────────
- 嚴格區分：
  - 表現層（UI / View）
  - 業務邏輯層（Domain / Use Case）
  - 資料存取層（Repository / API / DB）
- 禁止業務邏輯直接寫在 UI、Controller 或 API Handler 中
- 每個模組需有單一責任與清楚的對外介面
- 命名必須可讀且具業務語意，禁止抽象或模糊命名
- 禁止重複邏輯，需主動抽象與共用

────────────────────────
【三、前後端契約與 API 設計】
────────────────────────
- API 行為需可預測，不可隱含副作用
- 請明確定義：
  - 請求格式
  - 回傳格式
  - 錯誤碼與錯誤語意
- 前後端需共享同一份資料模型或型別定義（概念層級即可）
- 禁止為了前端方便而破壞後端資料完整性

────────────────────────
【四、測試與驗證（不可省略）】
────────────────────────
- 核心業務邏輯必須具備對應測試
- 測試需涵蓋：
  - 正常流程
  - 邊界條件
  - 錯誤與例外情境
- 測試目標為「驗證行為與規則」，不是只驗證程式能執行
- 若測試成本過高，需明確說明風險與原因

────────────────────────
【五、錯誤處理、觀測性與除錯能力】
────────────────────────
- 所有失敗情境必須：
  - 可被記錄（logging）
  - 可被追蹤（request / trace id）
- 錯誤訊息需對使用者與開發者分層處理
- 禁止吞掉錯誤或只回傳模糊訊息
- 關鍵流程需具備足夠的可觀測性以支援維運

────────────────────────
【六、使用者體驗一致性（產品級）】
────────────────────────
- 相同行為在所有頁面必須有一致的操作模式與回饋
- 非同步操作需明確呈現：
  - 載入中狀態
  - 成功回饋
  - 失敗處理
- 禁止讓使用者猜測系統目前狀態
- 重要操作需防止誤觸與重複提交

────────────────────────
【七、效能與擴充性（預設假設會成長）】
────────────────────────
- 預設資料量、使用者數與流量會成長
- 禁止不必要的同步阻塞流程
- 避免 O(n²) 以上的非必要設計
- 不得為了短期開發速度犧牲長期效能
- 設計需保留未來拆分與橫向擴充空間

────────────────────────
【八、安全性與權限控制（不可假設信任）】
────────────────────────
- 所有請求需經過身份驗證與授權檢查
- 前端驗證僅為 UX，不可作為安全依據
- 不可暴露敏感資訊於前端或 API 回傳中
- 權限判斷需集中管理，不可散落於各處

────────────────────────
【九、變更風險與向下相容】
────────────────────────
- 任何可能影響既有行為的修改需明確標示
- API 與資料結構變更需考量向下相容性
- 不可在未說明風險的情況下引入破壞性變更

────────────────────────
【十、原則優先順序（發生衝突時）】
────────────────────────
1. 資料正確性與系統一致性
2. 可維運性與可理解性
3. 安全性
4. 效能
5. 開發速度

────────────────────────
【十一、輸出規範】
────────────────────────
- 若需求不完整、存在矛盾或設計風險，需先指出再進行實作
- 不可自行補齊未明確定義的業務規則
- 所有說明與輸出一律使用正體中文
```

### 專案生成過程發現紀錄

- 對於 Spec、Code 的 Transition diagram 是不同的
    - 對於 Code，每個 State 就是實際程式中的狀態
    - 對於 Spec，每個 State 對應的只是流程規劃中的狀態
        - 須考量到 Spec 沒定義時的預設作法
        - 需考量到 State 的拆分細度
            - 太精細有可能會和實作的結果有誤差
            - 太寬鬆測試流程可能無法正確涵蓋 Full Coverage
        - 需考量到錯誤處理的導向機制
        - 需考量網頁的特性，將 Page、Feature 的關聯統一規範，所以整個網站會是一個多個 transition diagram 結合的完整系統
            - 實作概念構想
                
                ```bash
                我想像中會是 page 之間狀態改變的頁面會是整個網站系統的 state 轉換 transition diagram 的主幹，做甚麼事會導致頁面狀態發生轉換，接著功能類的 state 轉換的 transition diagram 看起來就會是 page transition diagram 的子圖的那種感覺，因為功能必然是某個頁面下功能，所以總而言之功能、頁面狀態的個別 transition diagram 很重要，所有 transition diagram 彼此之間的關係也很重要，我希望我可以藉由 transition diagram 了解整個網頁，ex: 登陸頁面已管理員身分進去後，看到活動頁面，因為我現在是活動頁面的狀態，並且是管理員狀態，所以我發現有關連到一個活動管理的功能，接著我就可以延伸到這個活動管理的功能去根據他的 transition diagram 了解功能是怎麼運作的，甚至怎樣的操作會又導致頁面切換。 
                
                Page 是主狀態機（Macro State Machine）
                Feature / Function 是子狀態機（Sub State Machine）
                角色（Role）是 Guard / Context
                操作（Action）是 Transition Trigger
                
                用一句話定義
                👉「我現在在哪一個頁面狀態？
                在這個頁面狀態 + 我的身分下，
                我可以進入哪些功能狀態機？
                這些功能操作又會不會把我送去別的頁面？」
                
                # 正確的 Diagram 分層方式（很關鍵）
                ## Layer 1：🌍 Global Page State Machine（整個網站骨架）
                - 描述「頁面 ↔ 頁面」
                - Trigger 是：
                  - 導航
                  - 登入成功 / 失敗
                  - 權限檢查結果
                - 不處理細部功能邏輯
                
                ## Layer 2：📄 Page-level State Machine（單一頁面）
                - 描述「這個頁面下會發生什麼狀態轉換」
                - 包含：
                  - loading / error / ready
                  - 依角色不同顯示不同入口
                  - 會連結到子功能狀態機
                
                ## Layer 3：⚙️ Feature / Function State Machine（子圖）
                - 一定掛在某個 Page State 下面
                - 描述：
                  - CRUD
                  - 表單
                  - 狀態改變
                - 可以：
                  - 回到原頁
                  - 或觸發 Page transition
                ```

> 
- 評估數值說明
    
    > 利用 ESLint 、sonarjs/cognitive-complexity 根據 Abstract Syntax Tree（抽象語法樹）做分析
    > 
    - **Cyclomatic Complexity (迴路複雜度)**
        - 這段程式碼有多少種「走法」
        - 看分支數量
            - 只要看到 `if`, `for`, `while`, `case` 等分支，分數就加 1。
            - **目的：** 多少個 Test Case 才能覆蓋所有路徑。
    - **Cognitive Complexity (認知複雜度)**
        - 這段程式碼對人來說有多難讀？
        - 不只是看分支，更看重**巢狀結構**。
            - `if` 裡面還有 `if`，分數爆炸性增長
            - 只有 `if`，分數不高。
            - **目的：** 程式碼**維護**有多麻煩。
    - **Cyclo/Fn(含分支)**
        - 每一個 Function 平均有多少分支。數值越低表示每個 Function 邏輯越乾淨
    - **Cog/Fn(含巢狀結構)**
        - 每一個 Function 平均有多複雜的巢狀結構。數值越高表示 Function 越繞口、複雜
- 欄位項目
    
    > 主要目標是為了比較 “只用 Spec 資訊生成專案”、”使用 Spec + Transition diagram 資訊生成專案” 哪個做的比較好? Transition 資訊會幫助專案生成的更好嗎?
    > 
    - **Index** : 順序
    - **專案名稱** : 名稱
    - **總數量** : 該專案的總功能數
    - **[Func] Spec** : 在只有 Spec 的版本下，成功正確完成的功能數量 (功能實作完成也可正常使用)
    - **[Func] Spec+Transition** : 在使用 Spec 加上 Transition  資訊的版本下，成功正確完成的功能數量 (功能實作完成也可正常使用)
    - **[行數] S** : 在只有 Spec 的版本下，最後生成專案的總行數
    - **[行數] S+T** : 在使用 Spec 加上 Transition  資訊的版本下，最後生成專案的總行數
    - **[Task] S** : 在只有 Spec 的版本下，Speckit 流程中規畫出的實作 Task 總數量
    - **[Task] S+T**  : 在使用 Spec 加上 Transition  資訊的版本下，Speckit 流程中規畫出的實作 Task 總數量
    - **[Cyclo] S** : 在只有 Spec 的版本下，Cyclomatic Complexity 總分數是多少
    - **[Cyclo] S+T** : 在使用 Spec 加上 Transition  資訊的版本下，Cyclomatic Complexity 總分數是多少
    - **[CycloFn] S** : 在只有 Spec 的版本下，用來計算 Cyclomatic Complexity 的總 Function 數量是多少
    - **[CycloFn] S+T** : 在使用 Spec 加上 Transition  資訊的版本下，用來計算 Cyclomatic Complexity 的總 Function 數量是多少
    - **[Cyclo/Fn] S** : 在只有 Spec 的版本下，每一個 Function 平均有多少分支。數值越低表示每個 Function 邏輯越乾淨
    - **[Cyclo/Fn] S+T** : 在使用 Spec 加上 Transition  資訊的版本下，每一個 Function 平均有多少分支。數值越低表示每個 Function 邏輯越乾淨
    - **[Cog] S** : 在只有 Spec 的版本下，Cognitive Complexity 總分數是多少
    - **[Cog] S+T** : 在使用 Spec 加上 Transition  資訊的版本下，Cognitive Complexity 總分數是多少
    - **[CogFn] S** : 在只有 Spec 的版本下，用來計算 Cognitive Complexity 的總 Function 數量是多少
    - **[CogFn] S+T** : 在使用 Spec 加上 Transition  資訊的版本下，用來計算 Cognitive Complexity 的總 Function 數量是多少
    - **[Cog/Fn] S** : 在只有 Spec 的版本下，每一個 Function 平均有多複雜的巢狀結構。數值越高表示 Function 越繞口、複雜
    - **[Cog/Fn] S+T** : 在使用 Spec 加上 Transition  資訊的版本下，每一個 Function 平均有多複雜的巢狀結構。數值越高表示 Function 越繞口、複雜
    - **[Func] Diff** : S+T 減 S，所以數值越高表示有 Transition 的版本做的越好，功能完成度越高
    - **[Cyclo] Diff** : S 減 S+T，所以數值越高表示有 Transition 的版本做的越好，Transition 的版本有讓 Cyclomatic Complexity 會降低
    - **[Cyclo/Fn] Diff** : S 減 S+T，數值越高表示有 Transition 的版本做的越好，Transition 的版本有讓每個 Function 邏輯越乾淨
    - **[Cog] Diff** : S 減 S+T，數值越高表示有 Transition 的版本做的越好，Transition 的版本有讓 Cognitive Complexity 會降低
    - **[Cog/Fn] Diff** : S 減 S+T，數值越高表示有 Transition 的版本做的越好，Transition 的版本有讓 Function 越不繞口、不複雜
    - **[行數] Diff** : S 減 S+T，數值越高低需搭配其他數值查看
- 我目前觀察到的結果 (不過仍需分析)
    - 對於功能完程度，有 Transition 的版本確實做的更好，功能不僅有完成且可正常使用，完程度更高
    - 對於複雜度的兩項指標 **Cyclomatic Complexity、Cognitive Complexity**，雖然有些許正相關看起來像是有 Transition 的版本可以降低數值，但趨勢不明顯
        - 我的觀點 : 即使多了 Transition 的資訊，產生的結果也不會因此變得複雜，在提升功能完整度的同時維持程式的複查度
    - 對於程式行數，有沒有 Transition 的版本看起來當不影響
        - 我的觀點 : 即使多了 Transition 的資訊，產生的程式碼也不會因此爆多、龐大，在提升功能完整度的同時就程式碼依然乾淨，有些專案程式碼甚至還比只有 Spec 的版本少，如此可以證明有 Transition 的版本甚至可以更好的去幫助 AI 規劃程式架構以及功能設計，減少程式碼的同時提升功能實作的完整度。

        ```bash
        export const PATH_NARRATOR_PROMPT = `【系統角色】
        你是 UI Path Narrator。你的任務是把「整條 path」轉成可執行、可驗證、且能逐 transition 推進的敘事指令，提供給 operator-loop 使用。
        
        【核心目標】
        1. 為整條 path 產出一份總體 narrative，說明這條 path 在測什麼、應如何推進、何時可視為完成。
        2. 為 path 中每個 transition step 產出一份 transition narrative，讓 operator-loop 能逐步執行與驗證。
        3. validations 必須能被觀測、可落地、且優先來自 diagram transition、connector、或 entry validation，而不是憑空想像。
        
        【不可違反的規則】
        1. 只能輸出合法 JSON，不可輸出 markdown、說明文字、註解、code fence。
        2. 不可改寫 path 結構，不可新增、刪除、重排 steps。
        3. 每個輸入 step 都必須在輸出 transitions 中出現一次，且以 stepId 對齊。
        4. 每個 transition 至少要有 1 個 validation；若原始資訊不足，仍要補出保守但可驗證的最小條件。
        5. validation.type 只能使用：url-equals、url-includes、text-visible、text-not-visible、element-visible、element-not-visible、network-success、network-failed、semantic-check。
        6. validations.id 在同一個 transition 內必須唯一，且應盡量穩定對應 edgeId。
        7. summary 必須精簡、可讀、聚焦結果；taskDescription 必須可執行，描述要做到什麼與何時算完成。
        8. executionStrategy 必須描述整條 path 的執行策略，而不是單一步驟的細節。
        9. 不可輸出與 path 無關的測試、驗證、或跨路徑假設。
        10. 不可輸出 terminationReason、decision、functionCalls；這些屬於 operator-loop，不屬於 narrator。
        11. 不可把缺少立即前提寫成「必然失敗」；operator-loop 可能先做有限的探索性動作來建立前提，再完成同一個 transition。
        
        【推導原則】
        1. 先理解 path.semanticGoal 與 path.name，再結合每個 step 的 from、to、summary、semanticGoal。
        2. 優先使用下列來源推導 validation：
           - step.validations
           - 對應 diagram transition.validations
           - 對應 connector.validations
           - transition.intent.summary
           - diagram.meta.entryValidations
        3. 若以上來源不足以形成可執行條件，再補保守的 semantic-check。
        4. semantic-check 必須仍然具體，例如「成功進入會員首頁且可繼續執行下一步」，不要只寫「成功」。
        5. 不要把多個觀察混成一條模糊 validation；可拆成多條明確條件。
        
        【整條 path narrative 要求】
        1. narrative.summary：一句話描述這條 path 的核心目的。
        2. narrative.taskDescription：描述整條 path 的操作與完成標準，讓 operator-loop 知道這是一條連續流程，不是互相無關的點擊集合。
        3. narrative.executionStrategy：描述執行策略，例如是否應維持同一 session、何時應優先看 URL、畫面文字、權限狀態，以及何時可安全 advance。
        4. 若 path.actorHint 存在，executionStrategy 與各 step 的 taskDescription 必須把它視為此 path 的唯一執行身分依據。不要要求 operator-loop 在執行時重新猜角色；若流程需要不同角色，代表 planner 應拆成不同 path。
        5. validation fail 或 pending 不應被描述成必然中止條件；只要 UI 流程仍可操作，就應標記 validation issue 並繼續執行，而不是把它寫成 path 必須停止。
        6. executionStrategy 應允許 operator-loop 在同一 session 內做「有限且與主線直接相關」的探索性動作，用來建立立即前提、修正偏離、或補足證據；但不可暗示它應任意擴散探索。
        
        【transition narrative 要求】
        ```

        - 候選的作法
            - 最有效率 :
                
                設定起點、終點，是否能在有限數量 N 下快速達成轉移覆蓋
                
            - Random Walk :
                
                從起始點開始，隨機選擇下一個轉移，直到到達終點或死胡同。重複 N 次。
                
            - 貪婪覆蓋演算法 :
                
                用 N 條路徑達到最大覆蓋，應該選哪幾條。
                
            - Ant Colony Optimization (ACO)    **(Random Walk 加上策略的概念)**
                1. 派出 N 隻螞蟻從起始點出發。
                2. 螞蟻走過的邊會留下「資訊素」，但這裡的機制可以反過來：螞蟻傾向於走「資訊素少」的邊（代表沒被測試過）。
