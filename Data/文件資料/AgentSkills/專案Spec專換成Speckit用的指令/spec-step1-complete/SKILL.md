## 目標
把「粗略需求」擴展為完整 Spec。除了補齊缺漏，還必須主動思考該類型網站**應有的功能、角色、流程與規則**，將必要項目補全，確保規格完整可實作。
完整 Spec 必須包含以下六段：
- 產品目標
- 使用者角色定義
- 使用者流程
- 功能需求
- 非功能需求
- 資料模型

## 輸入定義（必須包含）
- 使用者提供的 spec 草稿（可不完整）
- 若有既有名詞（頁面名稱、角色、狀態），必須沿用並補齊

## 輸出位置（必須寫檔）
- 將結果寫入工作目錄：/outputs/step1-spec.md
- 回覆中僅提供完成訊息與檔案連結

## 輸出格式（必須完整）

# 任務 Spec：<產品名稱>
<英文或副標>

### 4.4 主要頁面需求
此段不可只列「通用網站模板」的頁面名稱，必須**依使用者 spec 中實際出現的功能/流程推導**出本產品需要的頁面。

必須包含：
- 頁面清單（Page Inventory）：頁面名稱、用途、URL/route（若有）
- 各頁面責任（Page Responsibilities）：每頁要解決什麼問題、主要元件/區塊
- 主要 CTA 與互動（Primary CTAs/Interactions）：例如新增/編輯/刪除/切換月份/匯出等入口放在哪一頁，避免與 Header 重複
- Page-level 狀態：至少描述每個主要頁的 Loading / Error / Empty（若適用）

常見但不一定存在的頁面（僅作提醒，不可硬套）：
- 清單頁（List）
- 詳情頁（Detail）：若 spec 允許在清單頁用 modal/drawer 完成 CRUD，則可不需要獨立詳情頁
- 我的頁面（Profile/Settings）：若 spec 未要求個人設定，可不需要
- 管理後台（Admin）：若 spec 無 Admin 角色，則不應出現

#### 4.4.x 資訊架構與導覽（必填）
- 頁面清單（Page Inventory）：列出所有頁面名稱、用途、URL/route（若有）
- 路由存取控制（Route Access Control）：
	- 每個頁面標註：Guest 可否進入、User 可否進入、Admin（若存在）可否進入
	- 若不可進入：行為要明確（導向登入？顯示 403？顯示 404？）
- 導覽列/Header 規則（Navigation Visibility Rules）：
	- 依「登入狀態」與「角色」定義 Header 會出現哪些選項、哪些選項必須隱藏
	- 禁止用「顯示但點了才導登入」取代「不該出現」的 UI 規則（除非 Spec 明確允許）
- 全站共用元件責任（Layout Responsibility）：
	- 定義 Header/Sidebar 的共用行為（例如：登入/登出入口放在哪裡）
	- 定義每個頁面自己要放的 CTA（避免與 Header 重複）
	- 若某 CTA 可能出現在 Header 與頁面內，需規定優先級與去重規則（例如：Header 已有登入按鈕，頁面內不再顯示登入按鈕，改顯示提示文字或 disabled button）

## 5. 非功能需求（Non-functional Requirements）
- RWD
- Loading / Error / Empty
- 錯誤處理（401/403/404）
- 效能、安全、資料一致性
- UI 一致性：共用導覽/版面不重複顯示相同 CTA；同一動作的入口要一致。

## 規則
- 使用者若未提供欄位，需補齊合理欄位。
- 避免與輸入 spec 內容矛盾。
- 不可省略任何段落。
- 內容需「正確、完善、詳細」。
- 資料模型需能支援所有功能需求（例：報名、取消、名額計算）。
- 若存在時間/截止條件，需在流程與規則中明確化。
- 必須補上該類型網站常見且必要的功能/角色/流程（例：權限控管、錯誤處理、狀態機、資料一致性）。
- 必須補上「頁面導覽與可見性規則」（依登入狀態/角色），否則後續 state machine 會無法避免錯誤 UI。
- 必須補齊「資訊架構/頁面與導覽規則」，並主動檢查避免：
	- 未登入就顯示不該出現的導覽項（例如：我的頁面、管理頁面）
	- 同一個頁面因為 Header + Page 重複設計而出現重複 CTA（例如：兩顆登入按鈕）
